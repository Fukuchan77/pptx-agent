## 指摘事項

### CRITICAL レベル

#### CRITICAL: 環境変数のグローバル設定によるスレッドセーフティ問題

**ファイル**: [`src/pptx_agent/agents/llm_config.py:90`](src/pptx_agent/agents/llm_config.py:90)

**問題**: `WATSONX_PROJECT_ID` を `os.environ` に直接設定しており、マルチスレッド環境で競合状態を引き起こす可能性があります。

```python
if project_id:
    os.environ["WATSONX_PROJECT_ID"] = project_id
```

**推奨事項**:

- LiteLLMへproject_idを直接渡すメカニズムを検討
- 環境変数を使用する場合は、アプリケーション起動時に一度だけ設定し、実行時には変更しない
- 複数の異なるproject_idを使用する必要がある場合は、LiteLLMクライアントのインスタンスを個別に作成

#### CRITICAL: パストラバーサル検証のTOCTOU競合状態

**ファイル**: [`src/pptx_agent/validators/security.py:223-242`](src/pptx_agent/validators/security.py:223)

**問題**: シンボリックリンクのチェックと `resolve()` 呼び出しの間にTime-of-Check to Time-of-Use (TOCTOU) 競合状態が存在します。

```python
# NOTE (TOCTOU): There is an inherent time-of-check-to-time-of-use race
# between the exists()/is_symlink() checks here and the subsequent
# resolve() call.
```

**推奨事項**:

- ファイルディスクリプタベースのアプローチ (`open()` → `fstat()`) を採用
- サーバー/マルチテナント環境で使用する場合は必須の対応
- CLIツールとしては許容範囲だが、将来の拡張性を考慮して改善を検討

### HIGH レベル

#### HIGH: APIキーの不十分なマスキングによる情報漏洩リスク

**ファイル**: [`src/pptx_agent/config.py:197-219`](src/pptx_agent/config.py:197)

**問題**: API設定の検証失敗時にログメッセージが出力されますが、エラーメッセージ内にフィールド名が含まれる可能性があり、攻撃者に構成情報を与える恐れがあります。

```python
logger.warning(
    "Configuration validation failed for provider: %s",
    self.llm_provider,
)
```

**推奨事項**:

- ログメッセージをさらに汎用化
- プロバイダー固有の情報をログに含めない
- デバッグモードでのみ詳細情報を出力

#### HIGH: グローバル状態の使用によるテスト分離の問題

**ファイル**: [`src/pptx_agent/config.py:223-249`](src/pptx_agent/config.py:223)

**問題**: グローバルな `_config` 変数を使用しており、テスト間での状態の分離が困難になっています。スレッドロックは実装されていますが、並列テスト実行時に問題が発生する可能性があります。

**推奨事項**:

- 依存性注入パターンの採用を検討
- 各関数でConfigインスタンスを明示的に渡す
- `reset_config()` を各テストのセットアップ/ティアダウンで確実に呼び出す

#### HIGH: 同期ファイルI/Oによるブロッキング

**ファイル**: [`src/pptx_agent/main.py:125`](src/pptx_agent/main.py:125), [`src/pptx_agent/templates.py:46`](src/pptx_agent/templates.py:46)

**問題**: `read_text()` や `save()` などの同期ファイルI/O操作が非同期コンテキストで使用されており、パフォーマンスのボトルネックになる可能性があります。

**推奨事項**:

- 非同期ファイルI/O (`aiofiles`) の使用を検討
- 大きなファイルの場合、ストリーミング処理を実装
- I/O操作を専用のワーカースレッドで実行

#### HIGH: プライベートメンバーへの直接アクセス

**ファイル**: [`src/pptx_agent/pptx_wrapper/presentation.py:129,174`](src/pptx_agent/pptx_wrapper/presentation.py:129)

**問題**: python-pptxのプライベート属性 `_sldIdLst` に直接アクセスしており、ライブラリのバージョンアップ時に破綻するリスクがあります。

```python
slide_id_list = self._prs.slides._sldIdLst  # noqa: SLF001
```

**推奨事項**:

- python-pptxの公式APIで同等の機能が提供されるか確認
- カスタムヘルパー関数に隔離し、バージョン依存性を局所化
- ライブラリアップデート時の影響範囲を最小化

#### HIGH: 例外ハンドリングの過度な汎化

**ファイル**: [`src/pptx_agent/agents/utils.py:47`](src/pptx_agent/agents/utils.py:47)

**問題**: `except Exception` による包括的な例外キャッチが、予期しないエラーを隠蔽する可能性があります。

```python
except Exception as fallback_error:
```

**推奨事項**:

- 特定の例外タイプ（`ConnectionError`, `TimeoutError` など）を明示的にキャッチ
- 予期しない例外は再raiseして早期に発見
- エラーログに詳細なスタックトレースを含める

### MEDIUM レベル

#### MEDIUM: マジックナンバーの散在

**ファイル**: [`src/pptx_agent/agents/outline_generator.py:29-39`](src/pptx_agent/agents/outline_generator.py:29)

**問題**: コード内に説明のないマジックナンバーが散在しています（例: `10`, `20`, `2`, `4`, `8`）。

**推奨事項**:

- すべてのマジックナンバーを名前付き定数として定義
- `constants.py` に集約するか、モジュールレベルで定義
- 値の意味をドキュメント化

#### MEDIUM: 関数の複雑度が高い

**ファイル**: [`src/pptx_agent/agents/content_generator.py:174-256`](src/pptx_agent/agents/content_generator.py:174)

**問題**: `_generate_content_blocks()` 関数が複数の責務を持ち、複雑度が高くなっています（McCabe複雑度 > 10）。

**推奨事項**:

- 関数を小さな単位に分割（例: `_handle_chart_content()`, `_handle_table_content()`）
- 各コンテンツタイプのハンドラーを個別の関数に抽出
- Strategy パターンの適用を検討

#### MEDIUM: エラーメッセージの国際化対応不足

**ファイル**: 複数のファイル（validators, main.py など）

**問題**: エラーメッセージが英語のみで、日本語ユーザーに不親切です。一部のメッセージは日英両方ありますが、一貫性がありません。

**推奨事項**:

- エラーメッセージの国際化フレームワーク導入
- すべてのユーザー向けメッセージを多言語対応
- `output_language` パラメータに基づいてメッセージ言語を選択

#### MEDIUM: テンプレートパースのキャッシング不足

**ファイル**: [`src/pptx_agent/pptx_wrapper/presentation.py:24-40`](src/pptx_agent/pptx_wrapper/presentation.py:24)

**問題**: 同じテンプレートファイルが繰り返しパースされる場合、パフォーマンスが低下します。

**推奨事項**:

- テンプレートマニフェストのキャッシング機構を実装
- LRUキャッシュを使用して最近使用したテンプレートを保持
- メモリ使用量とのトレードオフを考慮

#### MEDIUM: ロギング設定の柔軟性不足

**ファイル**: [`src/pptx_agent/utils/logging_config.py`](src/pptx_agent/utils/logging_config.py)（ファイルは確認していないが、ログ出力から推測）

**問題**: ログレベルや出力先の動的な設定が不十分な可能性があります。

**推奨事項**:

- 環境変数でログレベルを制御可能にする
- JSON形式のログ出力オプションを追加（本番環境向け）
- ログローテーション設定を追加

#### MEDIUM: 型ヒントの不完全性

**ファイル**: 複数のファイル

**問題**: 一部の関数で戻り値の型ヒントが `Any` や省略されています。

**推奨事項**:

- すべての公開APIに完全な型ヒントを追加
- `mypy --strict` モードでの型チェックを実施
- pydanticモデルを最大限活用

#### MEDIUM: リトライロジックの設定可能性不足

**ファイル**: [`src/pptx_agent/agents/llm_config.py:202-227`](src/pptx_agent/agents/llm_config.py:202)

**問題**: HTTPクライアントのタイムアウト設定は構成可能ですが、リトライロジックがpydantic-aiに依存しており、細かい制御ができません。

**推奨事項**:

- カスタムリトライミドルウェアの実装を検討
- バックオフ戦略の設定を外部化
- リトライ可能なエラーと不可能なエラーを明確に区別

#### MEDIUM: メモリ使用量の最適化不足

**ファイル**: [`src/pptx_agent/pipeline.py`](src/pptx_agent/pipeline.py), [`src/pptx_agent/agents/content_generator.py`](src/pptx_agent/agents/content_generator.py)

**問題**: 大規模なプレゼンテーション生成時にメモリ使用量が増加する可能性があります。

**推奨事項**:

- スライドごとのストリーミング処理を検討
- 生成された中間データの早期解放
- メモリプロファイリングの実施

### LOW レベル

#### LOW: docstringの不統一

**ファイル**: 複数のファイル

**問題**: docstringのスタイルが統一されていません（Google vs NumPy vs reStructuredText）。

**推奨事項**:

- プロジェクト全体でdocstringスタイルを統一（推奨: Google Style）
- `pydocstyle` などのリンターを導入
- ドキュメント生成ツール（Sphinx）との整合性を確保

#### LOW: コメントの冗長性

**ファイル**: 複数のファイル

**問題**: 一部のコメントがコードの内容をそのまま繰り返しており、保守性を低下させています。

**推奨事項**:

- 「なぜ」を説明するコメントに焦点を当てる
- コードから明らかな内容のコメントは削除
- 複雑なビジネスロジックにのみコメントを追加

#### LOW: 変数名の一貫性不足

**ファイル**: 複数のファイル

**問題**: 類似の概念に対して異なる命名規則が使用されています（例: `template_path` vs `pptx_path`）。

**推奨事項**:

- プロジェクト全体で統一された命名規則を策定
- コーディングスタイルガイドを文書化
- リンターでネーミング規則を強制

#### LOW: 未使用のインポート

**ファイル**: 一部のファイルに存在

**問題**: 未使用のインポートが残っており、コードの可読性を低下させています。

**推奨事項**:

- `ruff` や `autoflake` で自動削除
- CI/CDパイプラインに未使用インポートチェックを追加
- プレコミットフックで自動クリーンアップ

#### LOW: 定数の重複定義

**ファイル**: [`src/pptx_agent/constants.py`](src/pptx_agent/constants.py), 各モジュール

**問題**: 一部の定数が複数の場所で定義されています（例: タイムアウト値、しきい値）。

**推奨事項**:

- すべての定数を `constants.py` に集約
- モジュール固有の定数は明確に文書化
- 定数の重複を検出する自動チェックを導入

#### LOW: テストカバレッジの可視化不足

**ファイル**: テスト全体

**問題**: コードカバレッジが測定されていない、または可視化されていない可能性があります。

**推奨事項**:

- `pytest-cov` でカバレッジレポートを生成
- CI/CDにカバレッジ測定を統合
- カバレッジバッジをREADMEに追加

#### LOW: 例外クラスの階層不足

**ファイル**: [`src/pptx_agent/validators/exceptions.py`](src/pptx_agent/validators/exceptions.py)

**問題**: カスタム例外クラスが平坦な構造で、エラーハンドリングの柔軟性が低下しています。

**推奨事項**:

- 例外クラスの階層を設計（例: `PPTXAgentError` を基底クラスに）
- カテゴリ別に例外を整理（ValidationError, SecurityError, RuntimeError など）
- 例外チェーンを適切に使用

#### LOW: デバッグ情報の不足

**ファイル**: 複数のファイル

**問題**: エラー発生時のコンテキスト情報が不十分な場合があります。

**推奨事項**:

- エラーメッセージに関連する変数の値を含める
- スタックトレースとともにアプリケーション状態を記録
- 構造化ロギング（JSON形式）の導入
